## 발견 / 결정사항

### D1 — `KpiService` cross-package 재사용 (public 메서드 한정, P0-S1 D1 함정 회피)
계약 §5 입력 = `KpiService.listMyAssignments(cycleId, employeeId)`. **결정: `ReviewService` 가
`KpiService` 를 생성자 주입해 public 메서드만 호출**. `MyKpiAssignmentResponse` (P0-S2 record) 의
`id`(=assignmentId) / `weight`(effective) / `achievementRate` 3 필드를 점수 산식 입력으로 사용. 순환
의존 없음 (review → kpi 단방향, kpi 는 review 미참조). private/패키지-프라이빗 호출 0 — P0-S1 D1
(cross-package private 호출 → upsert 위임) 함정 사전 회피. KpiService 내부 latestActual/effectiveWeight
계산을 그대로 신뢰 (중복 계산 없음).

### D2 — `kpi_score_detail` jsonb 이중 용도 (draft managerScore vs 동결 스냅샷)
계약 §5: "submit-manager 에서 동결 스냅샷 + kpiScore 산출 / PATCH(MANAGER_PENDING)의 itemScores 는 draft
저장만 — kpiScore NULL 유지" + "GET kpi-items 는 저장된 managerScore 를 merge". 별도 per-item 점수 테이블이
범위 외라, **`kpi_score_detail` 컬럼 1개를 상태별로 다르게 사용**:
- **MANAGER_PENDING (draft)**: PATCH itemScores 를 `[{assignmentId, managerScore}]` 부분 배열로 저장.
  `ReviewResponse.kpiScoreDetail` 노출은 **null** (동결 전), kpiScore 도 **null**. GET kpi-items 가 이
  draft 를 파싱해 live 계산에 merge.
- **MANAGER_SUBMITTED~ (동결)**: §1 풀 shape 11필드 배열로 동결. `ReviewResponse.kpiScoreDetail` 이 파싱
  배열로 노출, GET kpi-items 는 live 미계산하고 저장 스냅샷 그대로 반환.
- 노출 분기 = `SNAPSHOT_FROZEN` 상태 집합 (MANAGER_SUBMITTED/CALIBRATION/FINALIZED/APPEAL_*/ARCHIVED).
  P0-S1 D2 lib OutboxEvent JSON 패턴 (ObjectMapper `USE_BIG_DECIMAL_FOR_FLOATS`) 정합 — BigDecimal 정밀도
  보존. draft·snapshot 둘 다 `managerScore` 필드를 가져 `parseDraftManagerScores` 공용 파싱.

### D3 — §5 점수 산식 정확 구현 (clamp / 비-NULL 항목만 / 가중 합산)
- `autoScore = clamp(round(achievementRate × 100, 2), 0, 100)` — achievementRate null → null. 음수
  클램프 0, 100 초과 클램프 100 (테스트: ar 1.5 → raw 150 → 100).
- `itemScore = managerScore ?? autoScore`. managerScore 는 0~100 검증 (REVIEW_SCORE_OUT_OF_RANGE, null 허용).
- `kpiScore = Σ(itemScore × effWeight) / Σ(effWeight)` — **itemScore 비-NULL 항목만** 분자·분모 포함
  (점진 입력 허용). itemScore null (managerScore null + autoScore null = achievementRate null) 항목은
  분모에서도 제외 (테스트: A ar null·manager null 제외 → kpiScore = B 만으로 산출). 비-NULL 0개면 null.
  round 2 (HALF_UP). effWeight 는 P0-S2 계약상 NOT NULL이나 방어적 null skip 코드 유지.

### D4 — cycle 단계 게이트는 transition/submit 만, PATCH/delete/조회 무관
계약 §3 매트릭스: transition 4 + submit 2 각각 요구 cycle.status (SELF_REVIEW/MANAGER_REVIEW/CALIBRATION)
검증 (REVIEW_CYCLE_STAGE_MISMATCH — cycle FINALIZED/CANCELLED 포함 전부 이 코드로 통일). **PATCH(섹션
임시저장)·delete·조회는 cycle 단계 게이트 없음** — review 자체 상태(섹션 가드/DRAFT 한정/lock)만 검증.
KpiService 의 cycle-lock(FINALIZED/CANCELLED 쓰기 거부)과 다른 축 — review 는 cycle 의 *진행 단계*를
요구하지 *잠금*을 보지 않음 (review 생성·조회는 lock 무관). submit 전이는 transition endpoint 로 진입 시
REVIEW_INVALID_STATUS_TRANSITION (SELF_PENDING→SELF_SUBMITTED 등은 전용 endpoint 강제).

### D5 — finalGrade 밴드 = S_A_B_C_D policy 한정 (EvaluationPolicyRepository 1:1 lookup)
계약 §5: 밴드(S≥90/A≥80/B≥70/C≥60/D<60)는 `policy.ratingScale == S_A_B_C_D` 일 때만. **결정:
`EvaluationPolicyRepository.findByTenantIdAndCycleId(tenantId, cycleId)` (P0-S1 1:1 패턴)** 로 policy 조회
후 ratingScale 분기. ONE_TO_FIVE/ONE_TO_HUNDRED 또는 policy 부재 → finalGrade null (P0-S4 Calibration
보강 박제). finalScore 는 P0 단순 = kpiScore (MBO/competency/mra 가중 확장은 P1 — 컬럼만 박제 nullable).
kpiScore null 이면 FINALIZED 전이 자체 거부 (REVIEW_SCORE_INCOMPLETE) → finalScore null 도달 불가.

### D6 — score/mbo/competency/mra 컬럼 박제 (P0 미사용 nullable)
계약 §1: `mbo_score`/`competency_score`/`mra_score` 는 P0 미사용 박제 (P1 채움). entity 매핑 + DDL
numeric(5,2) nullable 유지 (P0-S2 D1 source_config 패턴 — validate 모드 DDL↔엔티티 정합, P1 추가 마이그
없이 활용). ReviewResponse 에는 노출 (계약 §6 shape 에 4 score 필드 포함) — P0 동작상 항상 null. set 경로
없음 (P1 도메인 진입 시 채움).

### D7 — Clock 주입 + ObjectMapper 인스턴스화 (P0-S1 패턴 정합)
`finalizedAt = Instant.now(clock)` 테스트 친화 — security/jobstructure Clock 주입 패턴. 2-생성자 (운영
`Clock.systemUTC()` 기본 + 테스트 고정 Clock). ObjectMapper 는 `EvaluationPolicyService` 동형으로 service
생성자에서 인스턴스화 (`USE_BIG_DECIMAL_FOR_FLOATS`) — 손상 jsonb 행은 graceful null/빈맵 폴백.
