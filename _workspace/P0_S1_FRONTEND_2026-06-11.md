## 발견·결정사항

### 결정 1 — 에러 코드 prefix 표기 차이
- 기존 `frontend-vite/src/api/error.ts::isPerformanceError` 는 `E97` prefix 점검 (Self/OKR 도메인 작성 당시 가정).
- BE 실측 (`PerformanceErrorCode.java`) + 본 슬라이스 사양 = **E98** prefix.
- 본 슬라이스는 `error.ts` 의 historical `E97` 가정을 건드리지 않고, **자체 매핑 헬퍼** `pages/cycles/errorMapping.ts` 로 E9804xxx 10종 매핑.
- 후속 정리 = `error.ts::isPerformanceError` 의 prefix 를 `E98` 로 정정 + 자매품 매핑 통합 (별도 슬라이스).

### 결정 2 — `@mantine/modals` 미설치
- 자매품 일부에서 `@mantine/modals.openConfirmModal` 패턴 사용.
- 본 저장소 `package.json` 에 `@mantine/modals` 부재 → **`@mantine/core::Modal` 인라인 confirm 패턴** 으로 대체 (각 CycleRow 가 `confirmDeleteOpen` state 관리).
- dependency 추가 회피 = 기존 패턴 정합 + 빌드 영향 0.

### 결정 3 — Spring Data Page envelope 재사용
- `selfEvaluation.ts::PageEnvelope<T>` 가 이미 정의되어 있어 그대로 import 재사용 (자매품 정합).

### 결정 4 — RQ key factory 명명
- 사양은 `cyclesKeys` 였으나 자매품 (`selfEvaluationQueryKeys`/`personalOkrQueryKeys`) 정합을 위해 **`cyclesQueryKeys`** 명명 채택. `all() / list() / detail(id) / policy(cycleId)` 4 메서드.

### 결정 5 — 상태 전이 매트릭스 FE 박제
- BE 상태기계와 동일 매트릭스 (`STATUS_TRANSITIONS`) 를 FE 에 박제 — Menu 에서 "현재 상태에서 가능한 다음 상태만" 표시.
- BE 추후 매트릭스 변경 시 두 곳 동기 필요 (Risk: 박제 동기 비용). OpenAPI codegen 도입 시 reachable-status enum 으로 자동화 검토.

### 결정 6 — 분포 입력 폼 조건부 렌더
- `policy.forcedDistribution` 입력은 `distributionMode ∈ {FORCED, HYBRID}` × `ratingScale === 'S_A_B_C_D'` 일 때만 렌더.
- ratingScale 이 `ONE_TO_FIVE` / `ONE_TO_HUNDRED` 인 경우 분포 키 자체가 다르므로 본 슬라이스 범위 밖 (사양 명시 = S/A/B/C/D 5 NumberInput).
- mode 또는 scale 변경 시 `forcedDistribution` 자동 초기화/제거.

### 잠재 후속
- 정책 미설정 행의 "정책 미설정" Alert 링크 클릭 → 정책 편집 모달 (구현 완료).
