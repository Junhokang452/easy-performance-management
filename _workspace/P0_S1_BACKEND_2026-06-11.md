## 8단계 상태기계 전이 표

```
PLANNED        → ACTIVE, CANCELLED
ACTIVE         → GOAL_SETTING, CANCELLED      (GOAL_SETTING 진입 시 policy 필수)
GOAL_SETTING   → MID_REVIEW, CANCELLED
MID_REVIEW     → SELF_REVIEW, CANCELLED
SELF_REVIEW    → MANAGER_REVIEW, CANCELLED
MANAGER_REVIEW → CALIBRATION, CANCELLED
CALIBRATION    → FINALIZED, CANCELLED
FINALIZED      → (종결)
CANCELLED      → (종결)
```

## 발견 / 결정사항

### D1 — Cross-package private 검증 메서드 호출 함정
초기 EvaluationCycleService.create 가 EvaluationPolicyService 의 `validateDistribution` / `validateRatingScaleKeys` 를 직접 호출했으나 두 메서드는 패키지-private (테스트 보조용). 컴파일 실패. **해소**: `policyService.upsert(saved.getId(), pr)` 가 내부적으로 두 검증을 이미 수행하므로 직접 호출 제거. 신규 cycle 은 PLANNED 라 lock 검증도 자동 통과.

### D2 — JSONB 매핑 패턴
lib `OutboxEvent.payloadJson` 의 `@JdbcTypeCode(SqlTypes.JSON) + String payload + columnDefinition = "jsonb"` 패턴 채택. BigDecimal 정밀도 보존을 위해 service 에서 ObjectMapper `USE_BIG_DECIMAL_FOR_FLOATS` 활성. Hibernate 6.x 자체 지원 — hypersistence-utils 의존 추가 불필요.

### D3 — default policy 자동 생성 경로
`Cycle.create` 시 `request.policy == null` 이면 `policyService.createDefault(tenantId, cycleId)` 호출 — HYBRID/S_A_B_C_D/false/false/3/null. policy 가 제공되면 `policyService.upsert` 위임 (분포·키·lock 검증 일원화).

### D4 — DELETE CASCADE 안전성
`fk_policy_cycle ... ON DELETE CASCADE` — cycle 자체가 PLANNED 일 때만 delete 가능하므로 policy 동반 삭제는 안전 (운영 중 cycle 의 policy 가 잘못 삭제될 위험 0).

### D5 — Pageable list + policy id lookup N+1
list 응답에서 각 cycle 행에 대해 policy id 를 lookup — 현재 슬라이스에서는 페이지당 20행 가정 단순 구현. P1 슬라이스에서 `findAllPolicyIdByTenantIdAndCycleIdIn(...)` batch 쿼리로 N+1 제거 권장 (현재는 LIVE 미진입이라 후속).

### D6 — update name 변경 시 중복 검증의 "본인 행" 예외
`validateDuplicateName` 에서 update 흐름은 본인 행 (excludeId) 의 기존 name 과 동일한 경우 허용. unique 위반 회피 + 사용자 UX.
