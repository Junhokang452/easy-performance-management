## 발견 / 결정사항

### D1 — `source_config` jsonb: entity 매핑 유지 (P0-S1 D2 패턴 채택)
계약 §1 "source_config jsonb 박제하되 P0 미사용 — 매핑 여부 P0-S1 D2 참고 후 판단". **결정: KpiNode 에
`@JdbcTypeCode(SqlTypes.JSON) String sourceConfig` 매핑 유지** (P0-S1 EvaluationPolicy.forcedDistribution
과 동일 lib OutboxEvent JSON 패턴). 근거: (1) Hibernate `validate` 모드(운영 가정)에서 DDL 컬럼이
엔티티에 매핑돼 있어야 스키마 정합 잡음 0 — 미매핑 nullable 컬럼은 validate 무해하나, 매핑 유지가 DDL↔엔티티
충실. (2) P1 진입 시 추가 마이그 없이 즉시 활용. P0 동작상 항상 null (set 경로 없음). DTO 노출은 안 함
(P0 계약 Response shape 에 source_config 미포함).

### D2 — 단일 패키지 단일 서비스 (4 entity aggregate 가족)
계약 권장대로 `com.easyperformance.domain.kpi` 단일 패키지 + `KpiService` 단일 서비스. 4 entity 가 강하게
결합(tree→node→assignment→actual FK 사슬 + cycle-lock 해석이 모든 쓰기에 cross-cut)이라 분리 시 cross-package
private 호출(P0-S1 D1 함정) 재발 위험. controller 도 단일(`KpiController`) — resource 루트가
cycle/tree/node/assignment/actual 5종이라 `@RequestMapping("/api/v1")` + 메서드별 full path (P0-S1
EvaluationCycleController 의 단일 base 패턴과 다름 — 본 슬라이스는 5 루트 횡단이라 의도적).

### D3 — cycle lock 해석 경로 (tree→cycle, 모든 쓰기 진입)
KpiNode/Assignment/Actual 은 cycle_id 컬럼이 없음 → 쓰기마다 `entity → tree.cycleId → EvaluationCycle`
역추적 후 `requireCycleWritable`(FINALIZED/CANCELLED 거부). create/update/delete + supersede 전 경로에
일관 적용. node update/delete 는 `node→tree→cycle`, actual supersede 는 `actual→assignment→node→tree→cycle`
4-hop. 읽기(list/get/my)는 lock 무관 (조회 허용).

### D4 — 형제 weight 합 가드 = "초과만 거부" (계약 §3 정책 충실)
`validateSiblingWeightSum`: 같은 tree + 같은 parent 형제의 weight 합(update 흐름은 본인 행 제외) + 신규
weight ≤ 1.0 + 0.001 일 때만 통과. **합 < 1.0 은 허용**(트리 점진 구축 — FE 가 childWeightComplete
뱃지로 가시화). KpiNodeResponse 의 `childWeightSum`(직속 자식 weight 합)/`childWeightComplete`(==1.0±0.001,
자식 없으면 false)/`assignmentCount`(본 노드 배정 수)는 service 가 응답 생성 시 계산. parentId null(루트
형제) 분기는 Spring Data 파생 쿼리가 null 파라미터를 `=` 비교로 처리해 미스매치 → `findSiblings` JPQL 로
`(:parentId IS NULL AND n.parentId IS NULL) OR (...)` 명시.

### D5 — latestActual = "아무도 supersede 하지 않은 살아있는 row"
append-only supersede 체인에서 latest 판정: `supersedesId IS NULL` 만으로는 부족(최초 row 가 정정돼도
NULL 유지). 정답 = supersedesId NULL 인 row 중 **이 row.id 를 supersedesId 로 가리키는 정정본이 없는** row
(`existsByTenantIdAndSupersedesId` 역참조 확인). asOfDate DESC, createdAt DESC 정렬 첫 유효 행. (P0 단순화:
정정 체인 깊이 1~2 가정 — 깊은 체인 follow 는 P1.) achievementRate = latestValue ÷ effectiveTarget
(scale 6, HALF_UP), target null·0 또는 value null → null.

### D6 — KpiActual UPDATE/DELETE 경로 부재 (repository·service·DDL 3중)
계약 §1 "UPDATE/DELETE 없음 (정정은 supersede 전용)" 준수: `KpiActualRepository` 는 save/find 만
(delete 변형 미선언), service 에 actual update/delete 메서드 없음, DDL 의 `uq_kpi_actual_supersedes`
UNIQUE 가 원본 1회 supersede 강제 → 재정정은 `KPI_ACTUAL_ALREADY_SUPERSEDED` 사전 차단(UNIQUE 위반 전).
actual source 는 create/supersede 모두 서버에서 `KpiActualSource.MANUAL` 고정 (요청 DTO 에 source 필드
없음).

### D7 — rm_employee/rm_org_unit 미수신 (P0-S6 전 plain UUID)
KpiTree.ownerOrgUnitId / KpiAssignment.employeeId / KpiActual.reportedBy 는 FK 없는 plain UUID
(`@Column(columnDefinition = "uuid")`, DDL 도 FK 제약 없음). MyKpiAssignmentResponse 는 employeeId
라벨 join 없이 effective weight/target + latestActual 만 — 사람 이름 표면은 P0-S6 rm 수신 후.
