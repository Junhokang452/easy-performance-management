## 발견 / 결정사항

### D1 — `ReportService.publish` 컴파일 결함 fix (본 세션)
직전 산출 코드에서 `publish` 의 누적 리스트가 `List<PerformanceReport> created` 로 선언됐으나 루프에서
`created.add(toResponse(...))`(반환형 `ReportResponse`)를 호출 → `incompatible types` 컴파일 에러 2건
(add 인자 + `ReportPublishResponse(...)` 생성자 인자). `created` 를 `List<ReportResponse>` 로 정정
(= `ReportPublishResponse.published` 필드 타입). 의미 변경 0 — 의도는 발행된 리포트의 응답 리스트.

### D2 — content 동결 = `ReviewService.getReview` public 재사용 (중복 산식 0)
§5 content 의 review 확정값(finalGrade/finalScore/kpiScore/managerComment + kpiScoreDetail 전체 사본)은
`reviewService.getReview(reviewId)` 의 `ReviewResponse` 에서 그대로 가져옴. `kpiScoreDetail` 은 ReviewService 가
이미 동결 스냅샷을 파싱한 `List<ReviewKpiItemResponse>` 이므로 **그대로 `content.kpiItems` 로 사본** (재계산
0 — P0-S3 동결 스냅샷이 SoT). cross-package private 0. mbo/competency/mra/nextAction = null 박제(P0/P1).

### D3 — 발행 시점 분포 = FINALIZED·finalGrade **비율** (CalibrationService 분포와 별개)
`computeFinalizedDistribution` = cycle 의 `status==FINALIZED` review 만 대상. **분모 = FINALIZED 전수**
(finalGrade null 행도 분모 포함) / **버킷 = {S,A,B,C,D} 만** (finalGrade null 또는 비-SABCD 는 분모만 포함·버킷
제외 — §5 박제). 비율 = count/total round 4(HALF_UP). FINALIZED 0개면 `{S..D: 0.0000}`.
**CalibrationService.getDistribution 과 중복 아님**: 거기는 `effectiveGrade`(finalGrade ?? bandGrade) 로
`{CALIBRATION,FINALIZED}` 전체를 **건수** 버킷 (UNRATED 포함). 여기는 **FINALIZED·finalGrade 만** **비율**
(건수 비노출 E9). 산출 시점·대상·단위·노출 모두 상이 → 별도 단순 구현(ReportService 내). 일괄 발행 시 분포는
1회 산출 후 모든 신규 row 에 공통 동결 (발행 묶음 내 일관성).

### D4 — active 판정 = supersede 체인 tail (KpiActual `superseded` computed 패턴)
- **active = 다른 행의 `supersedes_id` 로 참조되지 않은 행**. 단건 게이트(view/ack/supersede)는
  `existsByTenantIdAndSupersedesId(tenantId, reportId)` 1쿼리 → true 면 `REPORT_NOT_ACTIVE` 409 (KpiActual
  `KPI_ACTUAL_ALREADY_SUPERSEDED` 사전 차단 패턴 정합).
- **publish skip 판정**(`hasActiveReport`) + **my 조회**(`getMyReport`)는 review/cycle×employee 의 행 집합에서
  `supersedes_id` 집합을 빼 active 존재/단건을 판정 (체인 선형성 보장 — `supersedes_id` UNIQUE 로 active 는 최대 1개).
- **`superseded` computed**: `toResponse` 가 행마다 `existsBy...SupersedesId` 로 산출해 응답에 주입 (FE 가
  superseded Badge 렌더).

### D5 — publish 일괄 게이트 + skip 의미 (계약 §3)
- **게이트** = `cycle.status == FINALIZED` 한정 (`requireFinalizedCycle` → `REPORT_CYCLE_NOT_FINALIZED` 422,
  publish/supersede 공통). cycle 미존재 = `CYCLE_NOT_FOUND` 재사용.
- **대상** = `review.status == FINALIZED` **AND** active report 미존재. 그 외는 **skip 카운트(에러 아님)** — 단
  **FINALIZED 아닌 review 는 skip 카운트 외**(silently continue), **FINALIZED 이면서 이미 active report 보유한
  것만 skippedCount 증가**. 테스트 `publish_publishesFinalizedWithoutActiveReport_skipsRest` 로 검증
  (FINALIZED 무발행 1 + FINALIZED 기존 active 1 skip + MANAGER_SUBMITTED 1 count 외 → published 1/skipped 1).

### D6 — view/acknowledge 멱등 (mutable 예외 2 필드, 계약 §0-3)
- **view** = `viewed_at == null` 일 때만 set + save (이미 있으면 no-op — save 호출 0, 테스트 `times(1)` 가드).
- **acknowledge** = `!acknowledged` 일 때만 true + `acknowledged_at` + save (이미 true 면 no-op).
- 둘 다 **active 행 한정**(`requireActive` 선행 — superseded 면 멱등 분기 전 409). Clock 주입으로 set 시각
  테스트 친화 고정(FIXED `2026-06-30T00:00:00Z`).

### D7 — Clock 주입 2-생성자 (ReviewService/CalibrationService 패턴 정합)
`ReportService` 2-생성자 (운영 `Clock.systemUTC()` + 테스트 고정 Clock) → `publishedAt`/`viewedAt`/
`acknowledgedAt` set 시각 고정. `ReportJson` 은 Clock 불요(content 직렬화에 시각 미관여 — nextAction P1)이라
무인자 생성. 테스트는 `TenantSupport` fallback 테넌트(`...-001`) 사용(게이트 OFF — CalibrationServiceTest 정합,
TenantContext mock 불요).
